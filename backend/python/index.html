What I Delivered
  • Requirements: Gathered and refined business requirements through direct client conversations — translated business workflows into technical specifications
  • Backend Architecture: Django REST Framework with Model-View-Serializer pattern, using Django's ORM for database operations
  • API Design: RESTful endpoints for billing, staff management, inventory, and GST-compliant invoice generation with versioned URLs (/api/v1/)
  • Database: PostgreSQL schema designed for transactional workflows, invoice history, staff attendance, and financial reporting with proper indexing

  • Staff attendance tracking: Designed timezone-aware timestamp handling for accurate attendance records across client's work shifts

    TaskSphere

    Microservices-based task management — FastAPI, Docker, PostgreSQL

    Major Project
    Demo Source
    Problem: Monolithic task management apps lack independent deployability and fault isolation. Enterprise teams need user, task, and auth services operating independently. Existing solutions (Jira, Trello) are SaaS-only — not designed for teams that want self-hosted, modular task management with independent service scaling.
    Architecture
  • 3 independent microservices communicating via REST (httpx async client) for loose coupling
  • FastAPI with async/await pattern for high-concurrency I/O operations
  • API Gateway pattern implemented with Nginx reverse proxy for centralized routing and request filtering
  • Per-service database strategy enables independent scaling and fault isolation
  • Docker Compose for consistent multi-environment deployment with service dependencies
  • Pydantic models for request/response validation and OpenAPI schema generation
    • Engineering Decisions
  • Framework selection: FastAPI chosen over Django for this microservices project due to async support, lower overhead, and automatic OpenAPI documentation
  • Data consistency: Accepted eventual consistency across services rather than distributed transactions — used application-level validation and idempotent API design with unique request IDs
  • Inter-service communication: Chose synchronous REST calls over async messaging for simplicity at current scale; architecture supports migration to message brokers (RabbitMQ/Kafka)
  • API design: Versioned endpoints (/api/v1/) with pagination, filtering using query parameters, and OpenAPI documentation auto-generated by FastAPI
  • Database connection pooling: Configured SQLAlchemy connection pools with appropriate limits for each service's expected load
    • FastAPIPython 3.11+DockerPostgreSQLReactJWTSQLAlchemyPydanticNginx
    Outcome: Production-deployed microservices system with 3 independent services. Detailed schema, API docs, and deployment config on GitHub. Achieved sub-50ms response times for 90% of requests.

    StayNest

    Hostel & PG booking platform — Django REST Framework, React, PostgreSQL

    Major Project
    Demo Source
    Problem: Students need PG/hostel booking with gender-based filters, room-level availability, and short-term booking. Existing platforms (MagicBricks, Nestaway) target full-home rentals — lacking these specific workflows.
    Architecture
  • Django MTV architecture (Model-Template-View) with Django REST Framework for API layer
  • JWT authentication with djangorestframework-simplejwt and role-based permissions (User, Owner, Admin)
  • Cloudinary integration for image uploads and CDN delivery
  • PostgreSQL row-level locking (SELECT FOR UPDATE) in transaction.atomic() blocks to prevent double-booking at high concurrency
    • Concurrency Handling
  • Challenge: Two users booking the same room simultaneously could cause double-booking
  • Solution: Implemented database-level pessimistic locking using PostgreSQL's SELECT FOR UPDATE within Django's transaction.atomic() context manager
  • Implementation: Booking endpoint acquires row lock on property availability record before checking and updating status
  • Fallback: Added application-level validation and idempotency keys for booking requests
  • Testing: Simulated 50 concurrent booking attempts — zero race condition failures
